# AI 编程进入深水区后，为什么我们需要“经验资本层”？

> **一句话版**：这篇文章不想介绍一个新的 memory 工具，而是想讨论一个更基础的问题：当 AI Agent 开始持续参与真实研发交付后，团队如何把一次次任务中产生的判断、排障路径、约束和教训，沉淀成项目级、团队级、可审阅、可复用的工程资产。

## TL;DR

- **痛点 1：经验蒸发。** AI 帮我们完成了越来越多任务，但任务结束后，大量关键判断仍然散落在聊天记录、终端历史、个人脑子和 commit diff 里，下次遇到类似问题还要重新来过。
- **痛点 2：个人记忆不等于组织资产。** 很多工程经验属于项目和团队，不应该只绑定在某个人、某个模型账号或某段对话上下文里。
- **痛点 3：存下来不代表学会了。** 真正有价值的经验，需要被提炼、审核、激活、反馈和证明，而不是简单塞进一个向量库。
- **核心判断：** AI coding 的下一阶段，不只是让 Agent 更会写代码，而是让团队有能力把“做过”变成“下次更会做”。
- **expcap 的定位：** 它不是个人长期记忆，而是一层项目级、团队级、可共享、可审阅、可交付的经验资本层。
- **当前状态：** 它已经能跑通 `auto-start -> progressive-recall -> feedback -> auto-finish` 的基本闭环，并能通过 `status / doctor / dashboard` 观察资产质量、反馈闭环、候选队列和 Milvus/SQLite 健康状态。
- **当前 dogfood 数据：** 截至 2026-04-28 中午，本仓库已有 87 个资产、133 次 activation log；65/87 资产为 healthy；107/133 次 activation 被记录为 helpful；Milvus 在 82/133 次 activation 中贡献最终选中资产。
- **边界：** 它还不是成熟平台，也不是零治理成本的自动学习系统。现在更适合高频 AI coding、愿意边用边收敛的团队试用和共建。

## 0. 前言：我们不是缺更多上下文，而是缺可复用的工程经验

过去一年，大家谈 AI 编程，很多注意力都放在模型能力、Prompt 技巧、IDE 体验和 Agent 框架上。

这些当然重要。但当你真的把 AI 带进持续的软件交付里，很快会遇到一个更朴素的问题：

> 一个任务做完之后，这次经验到底留在了哪里？

多数时候，它留在聊天窗口里，留在某个人脑子里，留在一次性的终端历史里，或者散落在 commit、issue、文档、日志和代码注释之间。

短期看，这似乎没什么问题。AI 已经把这次任务做完了，效率确实提升了。但如果把时间拉长，问题会变得非常明显：

- 同一个坑会反复踩。
- 同一类判断会反复重做。
- 同一套排障路径会反复从零开始。
- 换人、换 Agent、换会话之后，项目经验很难被完整接续。
- AI 看起来参与了很多任务，但任务之间没有形成真正的复利。

这也是我开始认真思考 `expcap` 的原因。

我们真正缺的，不是再往上下文窗口里塞更多东西，而是需要一层新的工程基础设施：

> **把执行过程中产生的经验，变成项目级、团队级、可共享、可审阅、可交付的工程资产。**

我把这件事叫做：**经验资本化**。

## 1. 为什么不是“记忆”，而是“资本化”？

“记忆”这个词太容易让人误解。

它听起来像是在回答一个很简单的问题：

> 过去的东西有没有存住？

但真实工程里，我们关心的不是“有没有存住”，而是：

- 什么经验值得留下？
- 应该以什么形态留下？
- 它属于个人、项目，还是团队？
- 未来什么时候应该调出来？
- 调出来以后，能不能真的改变后续行为？
- 如果它错了、旧了、重复了，怎么被降级、替换或淘汰？

这些问题加在一起，已经不是 memory store 的问题，而是资产治理的问题。

所以我更愿意用“经验资本化”这个概念：

> **经验资本化，是指 Agent 将执行过程中产生的经验，经过留痕、提炼、估值、审核、激活和反馈，转化为未来可复用的工程资产，从而持续改变后续行为质量。**

这里最重要的一句话是：

> **把“做过”变成“下次更会做”。**

普通 Agent 做完一次任务，下次不一定更会做。有经验资本化能力的系统，至少应该做到：

1. 这次任务发生了什么，有证据留存。
2. 这次任务里什么值得复用，有结构化提炼。
3. 这条经验是否值得晋升，有审核和估值。
4. 下次相似任务出现时，它能被召回。
5. 召回之后是否真的有用，有反馈回写。
6. 多次反馈之后，资产状态能被更新。

这才是从“记住了”走向“学会了”的关键。

## 2. 个人记忆为什么不够？

很多人听到这里，第一反应会是：这不就是 AI memory 吗？

我觉得不是。

个人 memory 解决的是“某个助手如何更懂我”。这当然有价值，比如记住我的偏好、常用技术栈、表达习惯、个人工作方式。

但工程经验里有很大一部分，不应该属于个人。

比如：

- 这个仓库为什么这么组织目录？
- 某条链路历史上为什么不能这样改？
- 这个项目的健康检查应该看哪些信号？
- 这个服务的失败模式有哪些？
- 哪些经验已经被真实任务验证过，哪些只是一次性 workaround？

这些东西更像项目资产，而不是个人偏好。

如果它只存在于某个工程师的聊天记录里，它就很难被交接。<br>
如果它只存在于某个模型账号的私有记忆里，它就很难被团队复用。<br>
如果它只散落在 commit 和 issue 里，它就很难在下一次任务开始时被主动激活。

所以 `expcap` 的核心定位不是“让某个 AI 更懂你”，而是：

> **让项目和团队拥有自己的经验资产层。**

这也是它和个人记忆工具最根本的区别。

## 3. expcap 到底是什么？

如果用一句话解释：

> **expcap 是一个面向 coding agent 的项目级经验资本化系统。**

它不和 Codex、Claude Code 或其他助手的个人 memory 竞争。它服务的是另一层问题：

- 项目经验如何沉淀？
- 团队经验如何共享？
- 工程教训如何审核？
- 资产是否真的帮助了后续任务？
- 经验如何从一次性的 episode 变成可复用的 asset？

今天它已经有一条基本闭环：

```text
auto-start -> progressive-recall -> feedback -> auto-finish
```

大致对应：

- `auto-start`：任务开始前，先激活相关历史经验。
- `progressive-recall`：任务过程中，如果出现新错误、新文件、新阶段或 topic drift，再做增量召回。
- `feedback`：任务推进后，把这次 activation 是否真的有帮助写回系统。
- `auto-finish`：任务收敛后，把 trace、episode、candidate、asset 沉淀下来。

同时，它还有一组治理和观测入口：

- `status`：看资产数量、使用情况、反馈摘要、candidate queue。
- `doctor`：检查 SQLite、Milvus、feedback backlog、资产 proof coverage。
- `dashboard`：把 Effectiveness Snapshot 可视化出来。
- `review-candidates`：处理新提炼出来的候选经验。

底层上，我们现在采用的基本分工是：

- **Milvus**：核心语义召回层。
- **SQLite**：轻量状态索引、日志、反馈统计和降级支撑。
- **JSON asset store**：保存项目级经验资产的正文和结构化数据。

这里有一个很关键的设计判断：

> 召回不是最终裁决。召回只是把带来源、带证据、带风险提示的候选交给当前 Agent 或工程师判断。

这很重要。否则经验系统很容易变成另一个“看起来很权威但其实不可控”的黑盒。

## 4. 我们现在做到什么程度了？

这件事目前已经不是纯概念。

在我们自己的真实使用里，`expcap` 已经跑通了几类关键能力：

- 任务开始前能自动激活项目经验。
- 任务结束后能沉淀 trace、episode、candidate 和 asset。
- activation 是否有帮助，可以通过 `feedback` 写回。
- asset 可以有 `healthy / watch / needs_review / unproven` 状态。
- dashboard 可以看到 effectiveness snapshot。
- doctor 可以检查 Milvus、SQLite、candidate queue、feedback backlog 和 proof coverage。
- 一批 `unproven` 资产已经通过真实任务被验证成 `healthy`。

如果只说能力，还是有点像功能介绍。更有说服力的是当前这套系统自己的运行数据。

截至 2026-04-28 中午，在 `agent-experience-capitalization` 这个仓库的真实 dogfood 数据里：

- 已沉淀 `87` 个 assets、`106` 个 candidates、`133` 条 activation logs。
- Effectiveness Snapshot 当前是 `overall_score=77`，结论为 `healthy`。
- 资产质量是 `65/87 healthy`，另有 `4 watch`、`18 unproven`，没有 `needs_review`。
- Activation help 是 `107/133 helpful`，其中 `supported_strong=99`、`supported_weak=8`、`unclear=26`，当前 `pending=0`、`missing=0`。
- Milvus 在 `82/133` 次 activation 中贡献了最终选中资产，整体选中贡献是 `215/609`。
- 最近 14 天有 `324` 次写入，说明它不是静态 demo，而是在持续任务里被使用。
- SQLite 状态索引健康，Milvus Lite 本地召回可用；当前仍有一个低优先级 candidate queue 和一个 stale lock metadata 需要清理。

![expcap dogfood 数据快照](/Users/wuyue/Downloads/ChatGPT%20Image%202026年4月28日%2012_44_06.png)

这组数据不能被解读成“已经完成规模化验证”。它只来自当前项目的真实 dogfood，不是跨多个团队的大样本评测。

但它已经足够证明另一件事：

> `expcap` 不是停在概念层。它已经能在真实 coding-agent 工作流里持续产生、召回、反馈和验证经验资产。

这最后一点尤其重要。

因为一个经验系统最危险的状态，不是“经验太少”，而是“经验很多但没人知道哪些真的有用”。

如果系统只能不断提炼候选经验，却无法证明它们是否帮到了后续任务，那么它迟早会变成一个噪声仓库。

所以我们现在最看重的不是“已经存了多少资产”，而是：

> **哪些资产已经在真实任务中被激活，并获得了正向反馈。**

这也是 `review_status`、`temperature`、`activation_feedback_summary`、`unproven_validation_queue` 这些机制存在的意义。

它们不是为了让指标好看，而是为了回答一个非常工程化的问题：

> 这条经验，到底有没有在真实工作中产生帮助？

## 5. 价值在哪里？

如果只讲功能，`expcap` 很容易被误解成“又一个 memory 工具”。

但我认为它真正的价值不在功能列表，而在四层。

### 5.1 减少重复思考

团队不是缺“写代码的人”，而是经常在同类问题上反复重新判断、重新排障、重新收敛。

有些判断很贵。

比如一次 Milvus 召回质量问题，真正耗时的可能不是改几行代码，而是判断：

- 问题在召回层，还是 selection 层？
- 是路径配置错了，还是向量召回本身不准？
- 是运行环境问题，还是资产状态问题？
- 该先修代码，还是先补观测？

这些判断一旦只留在聊天记录里，下次就很难复用。

`expcap` 的第一层价值，就是把这类判断变成未来能被激活的资产。

### 5.2 让 AI 参与开始形成复利

很多团队现在用 AI，是单次提效。

今天快一点，明天也快一点，但今天的经验未必能改善明天的执行。

这就像每次都在高速奔跑，但路没有被修出来。

经验资本化想解决的是另一件事：

> 让每次任务结束后，系统都比之前更懂这个项目一点。

这个价值不是一装上就立刻爆炸式可见的，它更像复利。

一开始只是少踩一个坑。<br>
后来是少做一次重复排障。<br>
再后来，是新人或另一个 Agent 可以更快接手。<br>
最后，团队开始拥有一层不依赖某个人脑子的工程经验资产。

### 5.3 把个人能力变成组织资产

很多工程团队的真实瓶颈，不是没人会做，而是会做的人脑子里的经验太难转移。

AI coding 进一步放大了这个问题。

因为越来越多中间判断发生在“人与 Agent 的协作过程”里，而这个过程天然容易消失。

`expcap` 的组织价值在于：

> 把 AI 时代最容易流失的工程经验，重新变成项目和团队真正拥有的资产。

这件事如果成立，它的价值会超过单个工具本身。

因为它关系到团队能不能把 AI 使用经验从“个人技巧”变成“组织能力”。

### 5.4 让 AI coding 从玩具效率走向工程能力

如果 AI 只是帮我们更快写代码，它带来的主要是局部效率。

但如果 AI 参与研发之后，团队还能治理经验、证明经验、复用经验、淘汰经验，它带来的就是工程能力的升级。

这也是我认为 `expcap` 值得认真做的原因。

它不是在回答“这次能不能更快写完”。

它在回答：

> AI 参与之后，团队的工程经验能不能开始积累复利？

## 6. 它还不是什么？

为了避免把这件事讲飘，我觉得边界必须说清楚。

第一，`expcap` 不是万能记忆。

它不会因为存了很多东西，就自动变成一个什么都知道的系统。经验资产的质量，仍然依赖真实任务里的激活、反馈和审核。

第二，它不替代工程判断。

召回结果只是候选。当前任务是否真的适用，还需要 Agent 或工程师结合上下文判断。

第三，它还不是零治理成本的平台。

候选经验需要审核，重复资产需要处理，低价值经验需要压制，局部 workaround 不能随便晋升为团队真理。

第四，它现在仍然更像一个可工作的原型，而不是完整成熟的企业平台。

这不是自谦，而是很重要的使用边界。

如果我们把它包装成“已经完全成熟，大家直接用就行”，反而会伤害这件事的可信度。

更准确的说法应该是：

> 它已经足够用来验证方向、支撑高频 AI coding 团队试用，并开始产生真实复利；但它仍然需要在更多真实任务里打磨治理、去重、共享和组织级协作能力。

## 7. 谁现在适合用？

我会把适用人群说得窄一点。

更适合现在用的是：

- 高频使用 AI coding 或 coding agent 的团队。
- 已经开始遇到上下文断裂、重复排障、经验散失问题的团队。
- 愿意把经验资产当作正式工程对象来治理的团队。
- 愿意接受“边用边收敛”的团队。
- 有兴趣一起共建 project/team memory 基础设施的团队。

不太适合的是：

- 只做一次性脚本的人。
- AI coding 使用频率还很低的团队。
- 只想要个人偏好记忆的人。
- 不愿意做反馈和审核的人。
- 期待一个零维护、自动成熟系统的人。

一句话：

> **如果你的 AI coding 还停留在偶尔问答，可能暂时不需要它；如果你已经在让 Agent 持续参与交付，它迟早会碰到你正在痛的地方。**

## 8. 为什么值得现在讲？

一个自然的问题是：既然它还不是成熟平台，为什么现在就要写文章讲？

我的答案是：

> 因为经验资本化这种系统，不可能等“完全成熟”之后再开始用。

它的很多关键能力，必须从真实任务里长出来。

比如：

- 哪些经验值得晋升？
- 哪些资产会被反复证明有效？
- 哪些召回只是看起来相关？
- 哪些候选总是重复生成？
- 哪些经验看似合理，但长期没有真实帮助信号？

这些问题没法靠离线设计完全解决。

只有把系统放进真实研发任务里，让它被使用、被反馈、被审阅、被修正，才能知道下一步该优化什么。

所以这篇文章不是在宣布“我们完成了一个成熟产品”。

它更像是在说：

> 我们认为 AI 编程进入深水区之后，经验资本层会成为一块新的基础设施。我们已经跑出了一版能工作的原型，也已经看到真实价值。现在值得把这个方向讲清楚，并邀请更多真实场景一起验证。

## 9. 结语：把经验从消耗品，变成基础设施

AI coding 让代码生成变快了。

但如果每次任务的经验都随对话结束而消失，团队得到的仍然只是一次次局部效率，而不是长期工程复利。

我认为下一阶段真正值得做的事情，是把这些经验重新组织起来：

- 从聊天记录，变成项目资产。
- 从个人技巧，变成团队能力。
- 从一次性排障，变成可复用路径。
- 从“存住过去”，变成“改变未来行为”。

这就是 `expcap` 想做的事。

它还没有完成，但它已经跨过了“值得认真讨论”的线。

我们想做的，不是一个更会聊天的 AI。

我们想做的，是一层真正能积累工程复利的经验资本基础设施。
