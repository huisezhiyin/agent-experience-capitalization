# AI 编程进入深水区后，我们为什么需要“经验资本层”？

## 0. 前言：我们不是缺更多上下文，而是缺可复用的工程经验

过去一年，大家讨论 AI 编程，常见焦点几乎都在模型能力、Prompt 技巧、IDE 体验和 Agent 框架上。

这些当然重要，但当你真的把 AI 带进持续的软件交付里，很快就会碰到一个更基础的问题：

> 一个任务做完之后，这次经验到底留在了哪里？

多数时候，它留在了聊天记录里，留在某个人脑子里，留在一次性的终端历史里，或者散落在 commit、issue、文档、日志和代码注释之间。

这会导致一个很现实的结果：

- 同一个坑会反复踩。
- 同一类判断会反复重做。
- 同一套排障路径会反复从零开始。
- AI 看起来参与了很多任务，但任务之间没有形成真正的复利。

所以我们真正缺的，不是再往对话窗口里塞更多上下文，而是需要一层新的基础设施：

> **把执行过程中的经验，变成项目级、团队级、可共享、可审阅、可交付的工程资产。**

这就是我们做 `expcap` 的出发点。

## 1. 我们在解决什么问题？

如果把问题说得更短一点，其实只有一句话：

> **Agent 如何把“做过一次”变成“下次更会做”。**

很多所谓的 AI 记忆系统，本质上只是“存住过去”。

但工程场景里真正重要的问题不是“有没有记住”，而是：

- 什么经验值得留下？
- 应该以什么形态留下？
- 应该属于个人、项目，还是团队？
- 下次什么时候该调出来？
- 调出来以后，能不能真的改变执行行为？

如果这些问题没有回答，再大的上下文窗口、再长的聊天记录，也只是“堆积”，不是“沉淀”。

所以我们更愿意把这件事叫做：

> **经验资本化。**

不是 memory，不只是 learning，而是把经验变成资产，并让资产反过来影响未来执行。

## 2. 为什么不是“个人记忆”，而是“经验资本层”？

这是 `expcap` 和很多个人记忆方案最不同的一点。

我们不认为工程经验的最佳归宿，是某个模型账号下面的私有记忆。

原因很简单。

第一，工程经验天然具有项目归属。

很多知识不是“我个人喜欢怎么做”，而是：

- 这个仓库的目录约定是什么
- 这条链路历史上踩过什么坑
- 这个服务的健康指标怎么判断
- 这个项目里什么算可复用经验，什么只是临时 workaround

这些东西更像项目资产，而不是个人偏好。

第二，真正有价值的经验应该可转移。

如果一段经验只能被某一个人、某一个模型、某一个对话窗口调用，那它的组织价值非常有限。真正有价值的经验，应该能够跨：

- 人
- Agent
- 机器
- 仓库环境
- 模型实现

第三，工程经验需要治理，而不仅是存储。

不是所有经验都应该长期保留，不是所有 lesson 都应该直接晋升为“知识”。经验需要经历：

- 留痕
- 提炼
- 审核
- 激活
- 反馈
- 证明
- 衰减或淘汰

这就决定了它不是一个简单的“记忆容器”，而更像一层经验治理系统。

所以 `expcap` 的核心定位不是“帮某个 AI 记住更多”，而是：

> **为项目和团队构建一层可治理的经验资本层。**

## 3. expcap 具体在做什么？

如果只看实现层，`expcap` 做的事情可以概括成一条闭环：

> `save -> get -> activate -> feedback -> review -> prove`

更具体一点，今天它已经包含这些能力：

- `auto-start`：任务开始前，自动激活相关经验
- `progressive-recall`：任务进行中，在真正出现新信号时做增量召回
- `feedback`：把 activation 是否真的帮到了任务，写回系统
- `auto-finish`：任务收敛后自动沉淀 trace、episode、candidate、asset
- `status / doctor / dashboard`：审阅使用量、反馈闭环、候选队列、Milvus/SQLite 健康与资产质量
- `candidate review`：把经验从“提议”推进到“审核后的资产”
- `Milvus + SQLite`：Milvus 负责核心语义召回，SQLite 负责轻量状态索引、反馈统计和降级支撑

也就是说，`expcap` 不是只做存储，也不是只做召回。

它真正想建立的是：

1. 任务前有激活
2. 任务中有增量召回
3. 任务后有沉淀
4. 资产本身有健康状态
5. 激活结果有帮助反馈
6. 经验能在真实任务里被证明，而不是永远停留在“看起来不错”

## 4. 它现在做到什么程度了？

这件事现在已经不是纸上设计了。

从内部真实运行结果看，`expcap` 已经至少跨过了“概念 demo”这条线：

- 已经有完整的 `save/get/log` 闭环
- 已经能在任务开始前自动激活经验
- 已经能在任务进行中只基于新信号做 delta recall
- 已经能对 activation 写回 `supported_strong / supported_weak / unclear`
- 已经能根据真实反馈更新资产的 `review_status` 和 `temperature`
- 已经能审阅 `healthy / watch / unproven` 资产分布
- 已经能用 Milvus 做核心语义召回，用 SQLite 做状态索引与治理支撑

更重要的是，我们已经不只是在“生成资产”，而是在持续验证资产。

这意味着系统里存在一条很关键的差异：

- 有些资产只是被提炼出来了
- 有些资产已经在真实任务里多次被命中并证明有效

这条差异非常重要。因为一个系统如果只能“提炼经验”，却不能“证明经验”，它最终还是会变成一个噪声越来越多的仓库。

而 `expcap` 当前最重要的进展，恰恰在于它已经开始有能力区分：

- 哪些资产只是看起来合理
- 哪些资产已经在真实任务里形成了帮助信号

这说明它开始从“经验收集器”向“经验资本系统”转变。

## 5. 它还没有做到什么？

这部分必须说清楚，否则文章会显得太像宣传。

第一，它不是万能记忆。

它不会因为存了很多东西，就自动变成一个什么都知道、什么都能判断对的系统。经验资产的质量，仍然依赖真实任务中的激活、反馈和审核。

第二，它不替代人的判断。

召回结果只是候选。资产是否适用于当前任务，仍然应该由人或当前 Agent 结合上下文做最终判断。`expcap` 提供的是带来源、带证据、带风险提示的候选层，而不是终局裁决器。

第三，它仍然有治理问题要解决。

比如：

- 低价值候选如何自动压制
- 重复资产如何去重或合并
- 局部 workaround 如何避免被误晋升
- 跨项目共享时，如何保证不把局部经验误当组织真理

第四，它离“零成本自动成熟”还很远。

如果一个团队期待的是：装上以后不用管，系统自己就会越来越聪明、越来越干净，那现在显然还不是这个阶段。

今天的 `expcap` 更接近：

> **一个已经能工作、已经有复利趋势、但仍然需要真实使用来继续打磨治理能力的原型系统。**

## 6. 那它现在能不能用？

我的判断是：**能用，但不是所有人都适合现在就用。**

更适合的场景是：

- 正在高频进行 AI coding / agentic coding 的团队
- 愿意把“经验资产”当作正式工程对象来治理的团队
- 需要项目级、团队级经验转移，而不是只靠个人聊天记录的团队
- 能接受“边用边收敛”，而不是要求一步到位完美成品的团队

不太适合的场景是：

- 只写一次性脚本或极短期实验的人
- 并不打算做经验治理，只想多存点历史记录的人
- 期待零配置、零运营成本、零反馈成本的人
- 还没有真实任务频率，无法形成验证闭环的团队

换句话说，`expcap` 不是“人人现在都必须上”的工具。

但对于已经进入 AI 编程深水区、开始被上下文腐烂、经验散失、排障重复劳动和团队协作断层困扰的人来说，它已经足够值得开始用。

## 7. 为什么要现在开始，而不是等它“完全成熟”？

这是最容易被误判的地方。

很多人会直觉地想：

> 等它功能再完整一点、再自动一点、治理再成熟一点，我再用。

这听起来合理，但对经验资本系统来说，其实不完全成立。

因为这种系统最重要的能力，不是靠空想长出来的，而是靠真实任务中的闭环长出来的。

比如：

- 什么样的经验真正值得晋升
- 哪类资产最容易形成帮助信号
- 哪些召回虽然命中了，但并不真正有用
- 哪些候选总是重复生成，说明治理逻辑需要改
- 哪些资产长期 `unproven`，说明题目本身就不适合资产化

这些都不是离线设计能完全推出来的，必须在真实任务里被激活、反馈、审阅和修正。

所以 `expcap` 的成熟路径不是：

> 先做完，再开始用

而更像是：

> 先在真实任务里跑起来，再让“使用本身”推动它成熟

这也是我们为什么愿意现在就把它拿出来讲，而不是等到一个看起来更“完整”的时间点。

## 8. 我们真正想做的，不是一个更会聊天的 AI

如果只从表面看，`expcap` 像是一个围绕 Agent 的经验系统。

但更深一层，它想解决的其实是：

> 在 AI 参与软件交付之后，工程经验应该如何被保存、证明、治理和转移。

这件事今天还没有形成行业共识。

很多系统还停留在：

- 把历史对话当经验
- 把向量召回当经验复用
- 把“记住更多”当成“学会更多”

而我们的判断是：

> 经验真正有价值，不在于它被存住了，而在于它能不能被证明、被治理，并持续改变未来行为质量。

这也是为什么我们更愿意把 `expcap` 称为一层“经验资本层”，而不是一个“长期记忆工具”。

它的目标不是让 AI 更像一个记性很好的人。

它的目标是让项目和团队，开始拥有一种新的工程复利基础设施。

## 9. 结语：把经验从消耗品，变成基础设施

传统软件工程里，代码是最核心的产物。

而在 AI 参与编程之后，代码当然仍然重要，但另一个同样重要、却常被忽略的对象开始浮现：

> **经验本身。**

如果经验只是留在聊天窗口里，它就是消耗品。

如果经验能够被留痕、提炼、审核、激活、反馈和证明，它才会逐渐变成资产。

而一旦经验开始资产化，团队得到的就不只是“这次快一点”，而是：

- 下次更稳一点
- 换人更容易一点
- 换 Agent 更容易一点
- 换环境更容易一点
- 工程复利更真实一点

这就是我们理解的 `expcap`。

它还没有完成，但它已经跨过“值不值得认真做”这条线。

我们想做的，不是一个更会聊天的 AI。

我们想做的，是一层真正能积累工程复利的经验资本基础设施。
